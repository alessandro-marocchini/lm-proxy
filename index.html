01 — The Concurrency Bottleneck 01 — Il Collo di Bottiglia della Concorrenza

Local GPUs are powerful.
Parallel requests are deadly. Le GPU locali sono potenti.
Le richieste parallele sono fatali.

Running local Large Language Models (such as mistral-nemo or qwen3) on local development servers or workstation GPUs provides massive benefits: complete data privacy, zero call costs, and extremely low latency. However, these systems have a physical bottleneck: GPU VRAM.

L'esecuzione di Large Language Models in locale (come mistral-nemo o qwen3) su workstation o server di sviluppo offre enormi vantaggi: privacy assoluta dei dati, nessun costo di chiamata alle API e latenza bassissima. Tuttavia, questi sistemi si scontrano con un limite fisico invalicabile: la VRAM della GPU.

Unlike cloud APIs designed with auto-scaling cluster structures, a single local server running Ollama or LM Studio can crash or experience catastrophic slowdowns (due to KV cache collision or memory thrashing) when 2 or 3 heavy requests are run at the exact same millisecond.

A differenza delle API cloud progettate con infrastrutture a cluster auto-scalanti, un singolo server locale che esegue Ollama o LM Studio può andare in crash o subire rallentamenti catastrofici (a causa della collisione della KV cache o del memory thrashing) se 2 o 3 richieste pesanti arrivano contemporaneamente nello stesso millisecondo.

Moreover, developers frequently require an automated, transparent gateway that knows when to run a task locally (fast and free) and when to delegate it to the cloud (high token limits, complex reasoning), without exposing sensitive source code, corporate credentials, or company secrets to external endpoints.

Inoltre, gli sviluppatori hanno spesso bisogno di un gateway automatizzato e trasparente che sappia quando eseguire un task localmente (gratis e veloce) e quando delegarlo al cloud (limiti di token più ampi, ragionamenti complessi), senza esporre codice sorgente sensibile, credenziali aziendali o segreti industriali ad endpoint esterni.

The Goal L'Obiettivo

Deliver a zero-dependency Node.js proxy that intercepts all local LLM calls, queues them dynamically, scrubs them of secrets, and diverts excess load to cloud APIs the moment the local queue reaches saturation.

Fornire un proxy Node.js senza dipendenze esterne che intercetti tutte le chiamate ai modelli locali, le gestisca tramite code dinamiche, ne ripulisca i segreti aziendali e devii il carico in eccesso verso API cloud nel momento in cui la coda locale si satura.

"Workstation inference isn't about matching cloud scale. It is about protecting local slots and scaling outwards only when saturated." "L'inferenza locale su workstation non ha lo scopo di competere con la scalabilità del cloud. Il suo obiettivo è proteggere le risorse hardware locali e scalare all'esterno solo in caso di reale saturazione."

04 — Concurrency & Backpressure 04 — Gestione della Concorrenza e Backpressure

Dynamic queues.
Seamless cloud fallback. Code dinamiche.
Fallback trasparente sul Cloud.

The core feature of LM Proxy's VRAM protection is its dynamic queue coordinator. Instead of hardcoding performance thresholds, they can be configured at runtime from the dashboard without restarting the service.

La funzionalità chiave della protezione VRAM di LM Proxy è il coordinatore dinamico delle code. Invece di cablare le soglie di prestazioni a livello di codice, queste possono essere configurate in tempo reale dalla dashboard senza riavviare il servizio.

Step 01
A request arrives at the proxy. The Router evaluates the prompt and determines if the task target is Local (e.g. translation using mistral-nemo). Una richiesta arriva al proxy. Il Router valuta il prompt e determina se il target del task è Locale (ad esempio una translation che usa il modello mistral-nemo).
Step 02
The proxy checks the number of active local slots: activeCount. If activeCount < maxConcurrent (e.g. 2 slot), the request is processed immediately. Il proxy controlla gli slot locali attivi: activeCount. Se activeCount < maxConcurrent (es. 2 slot), la richiesta viene elaborata immediatamente.
Step 03
If all local slots are busy, the request is placed in the local queue. The queue depth increases: queue.length. Se tutti gli slot locali sono occupati, la richiesta viene inserita in coda locale. La profondità della coda aumenta: queue.length.
Step 04
Backpressure Check: If the queue depth exceeds the threshold maxQueueDepth (e.g. 3), and cloudFallbackOnQueueFull is enabled, the proxy diverts the incoming request to the Cloud Provider in real-time. Verifica Backpressure: Se la profondità della coda supera la soglia impostata maxQueueDepth (es. 3) e il fallback è abilitato (`cloudFallbackOnQueueFull`), il proxy devia la richiesta in tempo reale al Provider Cloud selezionato.
Step 05
The client receives a fast response from the Cloud, completely transparently, while the local GPU completes its slots without thrashing. Il client riceve una risposta veloce dal Cloud in modo del tutto trasparente, mentre la GPU locale completa i suoi slot in esecuzione senza subire crash.

05 — PII & NDA Anonymization 05 — Anonimizzazione PII e NDA

Scrubbing secrets.
Before they leave the LAN. Pulizia dei dati riservati.
Prima che lascino la rete locale.

Diverting overflow requests to a public cloud API (like Gemini or OpenAI) poses a major security compliance issue: private data leakage. To resolve this, LM Proxy integrates a local, multi-tiered Anonymizer.

Deviare le richieste in sovraccarico verso API cloud pubbliche (come Gemini o OpenAI) introduce un grave problema di conformità e sicurezza: la fuga di dati privati aziendali. Per risolvere questo problema, LM Proxy integra un sistema di Anonimizzazione a più livelli che opera interamente in locale.

Tier 01
Regex Filtering: Fast, deterministic scrubbing of classic patterns (Email addresses, Phone numbers, Credit Cards, IBANs, JWT tokens, AWS/Google API keys, IPv4/v6 addresses, and folder paths). Filtraggio basato su Regex: Rimozione veloce e deterministica di pattern noti (indirizzi e-mail, numeri di telefono, carte di credito, IBAN, chiavi JWT, chiavi API cloud, indirizzi IP e percorsi di cartelle di sistema).
Tier 02
Semantic Local LLM NER (Named Entity Recognition): If enabled, the prompt is evaluated locally by the base LLM to identify context-sensitive secrets: company names, client project codenames covered by NDA, employee names, and custom variables. Riconoscimento delle Entità Semantiche (LLM-NER locale): Se abilitato, il prompt viene esaminato dal modello locale per identificare informazioni sensibili dipendenti dal contesto: nomi di aziende esterne, nomi in codice di progetti protetti da NDA, nomi di dipendenti o variabili interne.
Tier 03
Token Splicing: Sensitive strings are replaced with safe, unique placeholders (e.g. [COMPANY_NAME_1], [API_KEY_1]). The mapping table is stored in local volatile memory. Sostituzione con Placeholder: Tutte le informazioni sensibili individuate vengono sostituite con identificativi sicuri e univoci (es. [COMPANY_NAME_1], [API_KEY_1]). La tabella di corrispondenza viene memorizzata nella memoria RAM volatile locale del server.
Tier 04
Prompt Dispatch: The clean, anonymous prompt is sent to the Cloud provider. Invio al Cloud: Il prompt, completamente ripulito e anonimizzato, viene spedito in sicurezza alle API del cloud provider esterno.
Tier 05
De-anonymization: Once the cloud returns the completion, the proxy reconstructs the response, replacing placeholders with original values before delivering the payload to the developer. De-anonimizzazione: Quando il cloud restituisce la risposta generata, il proxy la ricostruisce ripristinando i valori reali al posto dei placeholder prima di restituire il payload finale allo sviluppatore.

Key Findings & Insights Considerazioni e Dati Rilevati
Insight A
No GPU thrashing: Request #1 and Request #2 were processed in serial on the local GPU. Because only 1 model slot was active in VRAM, the workstation did not crash, overheat, or drop context tokens. GPU stabile e protetta: La Richiesta 1 e la Richiesta 2 sono state elaborate sequenzialmente sulla GPU locale. Avendo mantenuto attivo un solo slot di calcolo in VRAM, la workstation non ha subito surriscaldamenti, esaurimento di memoria o perdite di token di contesto.
Insight B
Anonymizer overhead: Request #3 (deviated to cloud) took 6.8 seconds. Why? Because the proxy executed the local LLM-assisted NER PII/NDA scrubbing first. The anonymizer step waited for active slots to clear, ensuring zero data leaks to public cloud APIs at the cost of waiting. If speed is preferred, disabling Tier 2 semantic scrubbing brings cloud fallback latency down to a mere 150 ms. Impatto dell'Anonimizzatore: La Richiesta 3 (deviata sul cloud) ha richiesto 6.8 secondi. Perché? Perché il proxy ha eseguito prima lo scrubbing semantico dei dati via LLM locale. Questa operazione ha atteso la liberazione dello slot locale, garantendo la sicurezza totale dei dati prima dell'invio esterno. Se la velocità è prioritaria, disabilitando lo scrubbing semantico (Tier 2) si riduce il tempo di instradamento al cloud a soli 150 ms.
Insight C
Predictable scaling: Without LM Proxy, Request #3 would have either crashed Ollama (403/500 Out of Memory) or queued behind Request #2, stretching its total wait time to over 15 seconds. By routing it to cloud, the developer gets an immediate response without workstation bottlenecks. Scalabilità prevedibile: Senza LM Proxy, la Richiesta 3 avrebbe causato il crash di Ollama (Out of Memory) o si sarebbe accodata dietro alla Richiesta 2, portando il tempo totale di attesa ad oltre 15 secondi. Con l'instradamento cloud il client riceve la risposta subito e in totale sicurezza.

Stored in plain JSON. Impostazioni modificabili a caldo.

The proxy's entire state and routing configurations are persisted in a single .router-config.json file. Every alteration made on the dashboard Vue UI is written directly to disk and reflected instantly in incoming calls.

L'intero stato del proxy e le regole di routing sono salvati in un unico file .router-config.json. Qualsiasi modifica effettuata tramite la dashboard grafica viene scritta direttamente su disco ed applicata istantaneamente a tutte le nuove richieste in ingresso.

Ingress and Egress Management Rules Regole di Gestione in Ingresso e Uscita
  • The Local Concurrency queue governs ONLY requests whose evaluated target is local. La coda di concorrenza locale gestisce ESCLUSIVAMENTE le richieste destinate ai modelli locali.
  • Requests explicitly mapped to cloud bypass the queue and execute asynchronously. Le richieste mappate direttamente sul cloud scavalcano la coda ed eseguono asincronamente.
  • Streaming responses (SSE) are fully supported during cloud fallback. Le risposte in streaming (SSE) sono pienamente supportate anche durante il fallback sul cloud.
  • If the cloud key is missing, backpressure fallbacks degrade gracefully to local queue wait times. Se la chiave cloud non è configurata, la richiesta di fallback rimane in coda locale degradando in modo sicuro.
08 — FAQ 08 — Domande Frequenti

Common questions. Domande frequenti e chiarimenti.

Does this require installing external dependencies? È richiesta l'installazione di dipendenze esterne?
No. LM Proxy is written entirely in native Node.js using core libraries (http, fs, crypto). This guarantees maximum startup speed, absolute compatibility with any server OS, and minimizes security vulnerabilities. No. LM Proxy è scritto interamente in Node.js nativo utilizzando solo moduli di sistema (http, fs, crypto). Questo garantisce un avvio fulmineo, massima compatibilità multipiattaforma e riduce a zero le vulnerabilità esterne.
How does the anonymizer affect latency? Qual è l'impatto dell'anonimizzatore sulla latenza?
Tier 1 regex scrubbing is practically instantaneous (<1ms). Tier 2 semantic LLM scrubbing adds a delay matching the speed of your local model (approx. 1-2s). If speed is critical, you can disable Tier 2 and rely only on Tier 1 regex. La rimozione di Tier 1 (tramite Regex) è istantanea (<1ms). La rimozione semantica di Tier 2 (via LLM) introduce una latenza legata alla velocità del modello locale (circa 1-2s). In casi ad alta priorità di velocità, è possibile disattivare il Tier 2 mantenendo il Tier 1 attivo.
Can I hot-swap models during live operations? È possibile scambiare i modelli a caldo durante le operazioni?
Yes. Using the Vue 3 control panel, you can alter active models, change API keys, and adjust queue depth thresholds. The changes are saved instantly and apply to all subsequent chat completion requests without restarting the Node server. Sì. Tramite la dashboard Vue 3 è possibile cambiare modelli attivi, chiavi API e limiti di concorrenza/coda. Le modifiche vengono salvate all'istante e si applicano a tutte le nuove richieste senza dover riavviare il server.
What local engines are supported? Quali motori di inferenza locali sono supportati?
Any local engine that exposes an OpenAI-compatible API. The proxy detects models via Ollama's local registry by default, but can be configured to map to LM Studio or LocalAI ports. Qualunque motore locale che esponga API compatibili con lo standard OpenAI. Il proxy rileva di default i modelli locali tramite il registro di Ollama, ma può essere puntato anche a porte di LM Studio o LocalAI.
